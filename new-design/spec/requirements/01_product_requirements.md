# 需求规格 · AI-Reader-V2(new-design)

> 逆向自 `app/` 代码与既有设计文档。描述系统**应当满足且当前已满足**的需求。

## 1. 产品目标

把中文长篇叙事文本(网文/小说)结构化为可浏览、可校验、可追溯的**七维度知识图谱**:
人物、物品、地点、组织(实体层)+ 场景、事件、时间(时序层)。用户上传 txt/zip,系统逐章
分析、跨章缝合,前端以图谱 / 阅读 / 时间线 / 场景多视图呈现,每个结论都能反查到原文出处。

核心信条(贯穿全部需求,不可削弱):**模型做判断 → 代码做确定性兜底校验 → 不确定项交人工**。

## 2. 功能性需求(FR)

### FR-1 多小说库
- FR-1.1 上传 `.txt` 或 `.zip`(zip 内多 txt 视作多文件),按小说名 slug 化独立存储。
- FR-1.2 重传同名小说返回 409,不覆盖。
- FR-1.3 列出全部小说,标注每本的 stage / 是否运行中 / 规则指纹是否过期(dirty)。
- FR-1.4 可改 meta(书名、作者、封面、标签、清洗规则选择、功能标签候选清单)。
- FR-1.5 可删除小说(运行中拒绝删除,返回 409)。

### FR-2 清洗与拆章(确定性)
- FR-2.1 按 `NOISE_PATTERNS` 删噪音行,按 `CHAPTER_PATTERNS` 切章。
- FR-2.2 规则可配:预制规则 + 用户自定义正则规则 + 用户预设组合;每本书可独立勾选。
- FR-2.3 `reclean` 用当前勾选规则重跑清洗+拆章(不分析),全局重排为 `input/<slug>/chNN.txt`。
- FR-2.4 改规则后正则非法返回 400;不可覆盖预制规则 id。

### FR-3 逐章七维度分析
- FR-3.1 每章按固定步骤跑:场景拆分 → 人物(识别+关系)→ 物品(抽取+关系)→ 地点(识别+关系)
  → 事件(父+子)→ 组织 → 归并+后处理。
- FR-3.2 场景拆分只在两类客观硬边界处切(地点物理移动 / 往事回放);不加合并类后处理。
- FR-3.3 人物/物品/地点均两 pass(先识别后关系),关系 pass 注入前一 pass 的候选清单(纯数字 id)。
- FR-3.4 事件两层:父事件(章级骨架)+ 子事件(场景级细节),子事件施事者可从父继承。
- FR-3.5 每章产出确定性后处理:全向图索引(`_graph`)+ 漏标疑点扫描(`_gap_suspects`),只报不改。

### FR-4 跨章缝合(确定性 + 模型复核 + 歧义留痕)
- FR-4.1 人物/物品/地点/组织跨章归一为全局实体(并查集 + provenance `members`)。
- FR-4.2 归并键须过 `_is_merge_key`:剔除代词/泛称/绰号/单字/「姓+通用职务」,防超级簇坍缩。
- FR-4.3 物品/地点的 `mentions`(原文定位短语)只入 `aux`,**绝不**作归并键或展示别名。
- FR-4.4 非高置信归并(仅别名重叠等)进 `ambiguities`,代码不硬合并。
- FR-4.5 缝合产出:个人时间线、跨人物同步点、全局关系(局部 id 映射到全局 id)。
- FR-4.6 启用 LLM 复核时,归并顺序铁律:封闭式清洗 → 精确同名合并 → 模型复核 → 缝合时间线;
  复核带磁盘缓存(已判词/对复用),仅新增词/对发起新判定。

### FR-5 只读浏览 API
- FR-5.1 端点:`summary / graph / events / chapters / reader/<ch> / dimension/<name> / node/<type>/<id>`。
- FR-5.2 读类端点支持 `?novel=<slug>`,缺省取最近上传的小说。
- FR-5.3 图谱含四类边:人物关系 / 地点关系 / 物品边 / 事件参与;事件作为一等节点进图。
- FR-5.4 节点详情返回全部原文出处(锚点在各章原文反查定位)。

### FR-6 任务控制与进度
- FR-6.1 异步分析(后台线程);进度落 meta.json,前端轮询 `progress/<slug>`。
- FR-6.2 软停:`pause`(章间阻塞,不打断进行中的章)/ `resume` / `stop`(停后续章,聚合已完成部分)。
- FR-6.3 断点续跑:已有 `_merged.json` 的章跳过。
- FR-6.4 单章失败隔离:不中断整本;完结时据实区分 `done` / `partial`(含失败/未跑章统计)。
- FR-6.5 僵尸态修正:meta 标活动态但进程不在运行 → 对外展示 `interrupted`。

## 3. 非功能性需求(NFR)

- NFR-1 **可靠性四道防线**(见 §4),根除模型幻觉,是最高优先级,任何改动不得削弱。
- NFR-2 **后端运行时零三方依赖豁免边界**:只读核心逻辑(`readonly.py`)纯标准库;合并后端
  (`server/main.py`)已用 Flask(经批准);前端 `web/`(Vite+React)豁免零依赖纪律。
  CI 结构校验依赖 `jsonschema`(开发依赖,不进运行时)。
- NFR-3 **可追溯性**:每个实体/关系/事件/场景结论都带锚点,可逐字反查原文。
- NFR-4 **可恢复性**:任务可中断续跑;global/ 原子提交防读到半成品。
- NFR-5 **隔离性**:生产部署(wangcai:8543)与平台配置、老版本(8011/8443)完全隔离。
- NFR-6 **性能留意点**:跨章并查集 O(n²),长篇全本需关注;冷启动 LLM 判定与章分析竞争 GPU,
  需先离线暖缓存再正式跑(见测试/部署规格)。

## 4. 可靠性红线(四道防线 · 对应 MODEL.md R1–R6)

1. **锚点校验**:所有 mention/alias/事件物品/场景首尾必须逐字出现在原文,否则剔除并记 `_validation`/`_anchor_miss`。
2. **id 引用**:Pass2 / 跨章 / 子事件只能引用已存在 id,机制上无法凭空造实体;悬空引用记 `_dangling_ref`。
3. **绝对时间纪律**:只认原文字面时间,无明确依据一律 `null`,绝不推算。
4. **歧义交人工**:跨章/脏名非高置信归并全进 `ambiguities`,代码不擅自终裁。

## 5. 约束(Constraints)

- C-1 模型调用必须 `format:"json"`、`think:false`、`stream:false`;temperature 场景 0.15 / 其余 0.12;
  整章 `num_ctx=49152`,事件管道默认 8192。
- C-2 提示词用 `prompts/` 下编号英文名(`01_..`),不得用历史中文名(会 FileNotFoundError)。
- C-3 候选清单注入提示词用纯数字 id(不加 C/I/L 前缀)。
- C-4 物品关系:一对一用 `part_of`,平级成套用 `set_group`。
- C-5 模型按任务适配:抽取类用 35b,场景边界判断用 27b(只改 `PASS_MODELS` 字典)。
- C-6 归并/校验/后处理代码必须纯确定性,不得引入二次模型调用(LLM 复核归并是显式例外,走独立模块 + 缓存)。

## 6. 已知局限(承认而非缺陷)

- 容器关系靠关键字黑白名单,词表外容器会漏标(README 已承认会过度标注)。
- 「往事夹叙同地点」「极密集时空跳转」章节场景边界有 ±1 摆动,是文本固有难度,不强加规则。
- 历史样本用旧 `events` 键,新流程用 `parent_events`/`sub_events`;`aggregate.py` 已兼容。
- 8543 外网可达性(云安全组/防火墙)未在文档中确认。
