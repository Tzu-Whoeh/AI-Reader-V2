# 人物-事件-在场-指代 — 探索结论(findings)

记录"如何从事件框架抽取/校验人物 + 在场判定 + 指代消解"这条线的关键结论。
**进行中的线**,已得出阶段性可用结论,正式接管线前先读这里。

## 背景:为什么探索这条线

独立人物 pass(`02_character_pass1`)在复杂文本上有错:周姐妹误合(ch44)、"老段落"黏连、
"克强"误抽、把组织(夜莺/别动队)当人物。设想用事件框架反过来抽/校人物。

## 1. 两条路径对比:事件抽人 vs 独立抽人

三章(ch44/doc3/doc4)× 2路径 × 3跑(数据见 path_comparison.json):

| 维度 | 甲:事件抽人(participants去重) | 乙:独立抽人(02_character pass) |
|---|---|---|
| 稳定性 | ⭐ 三章全部三跑一致 | ✗ 三章全漂移(6/6/2、名单/黏连变) |
| 精度 | ⭐ 高(抽到的几乎全对) | 中(混组织/泛指/黏连) |
| 召回 | ✗ 低(只抽动作参与者,漏配角如老赵/王凤滟) | ⭐ 高(主要角色基本全到) |
| 人/组织区分 | ⭐ 自带(只认动作者,组织被滤) | ✗ 分不清(夜莺/别动队当人) |
| 错误性质 | 稳定可预测(章首污染,可定位) | 随机(黏连/漏抽/噪声,难后处理) |

**结论**:甲=高精度核心锚(稳、自带人/组织过滤,但漏配角);乙=高召回候选池(全,但需清洗)。
二者**互补**,非二选一。理想架构:甲做骨干 + 乙补召回 + 差集过实体维度判别(27b)清洗。

**重要发现**:稳的错 >> 随机的错。甲的章首污染是稳定可定位的(只在与背景句相邻的第0个事件),
乙的"run3塌成2人"是不可预测的漏抽,后处理补不了。

## 2. 在场判定应解耦成独立 pass

"边抽事件边填 participants"会因"逐事件填槽"诱发就近抓取(章首背景句的人混入第0个事件)。
**把"在场判定"从事件抽取里解耦、单独做**,更稳更准:
- 独立在场判定(`scene_presence.txt`)在 doc4/doc3 全对,9场景8个三跑一致。
- 还额外召回了配角(doc3 司机老赵——甲漏的它抽到了)。

## 3. 在场判定按场景 type 路由不同 prompt

不同 type 的场景,"谁在场"的判定逻辑不同,用不同 prompt 分治:
- `内心独白/思绪段` → `scene_presence_inner.txt`:在场者=思考者本人,思绪里想到的人全排除。
- `现实叙述/动作` → `scene_presence.txt`(通用):正常判身体在场者。
- `回忆` → (待写)回忆里的人标为非当下在场。

pipeline 读场景 type 字段路由(类比 PASS_MODELS 按 pass 选模型,这是按场景类型选 prompt)。

## 4. viewpoint 字段解决思绪场景指代(关键创新)

**指代消解("他"是谁)不靠正则、不靠喂上下文猜,而靠场景切分时标注 `viewpoint`(视角人物)。**
- 场景拆分 prompt 加 `viewpoint` 字段(`01_scene_splitting_v9.txt`):标这段透过谁的视角叙述。
  实测:思绪段=思考者、对话段=客观或主视角、独处=当事人。9场景三跑全一致,准且稳。
- 回归:加 viewpoint 不破坏 v8 场景拆分(ch44/doc3 段数+cut_reason 与 v8 基线完全一致;
  doc4 的 loc/time 摆动是它自己的老边界问题,非 viewpoint 引入)。
- **闭环验证**:viewpoint=华剑雄 喂给思绪在场判定 → present 直接输出"华剑雄"(不再是未解析的"他"),
  thought_of 正确分流柳媚/黎子午/何小月/藤原香子,各带在场判断证据。三跑一致。

**优雅之处**:viewpoint 一次标注、多处受益(指代 + 情感分析"谁的情绪" + 叙事视角)。
思绪场景指代被 viewpoint 直接解决,独立指代消解 pass 只剩对话场景的"他/她"(有说话人轮替,好解)。

## 5. 试过但无效/否决的方案(别重走)

- **正则抓章首"就在…的时候"句**:否决,过拟合句式,换写法即失效。
- **"participants 必须在该事件 anchor_text 里"结构约束**:无效。中文叙事说话人名常在引号外,
  连续动作承前省略主语→把真在场者也误杀(doc3 约束后只剩柳媚)。证明"参与者∈anchor句"前提错误。
- **改 prompt 加"排除背景人"窄规则(v2)**:doc4 治好但 ch44 召回变多且不稳——扰动了 prompt 整体稳定性。
