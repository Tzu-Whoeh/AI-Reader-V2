# 格角色统一分流 + 事件时间线 — 探索结论

接续 entity_via_events 线。两大进展:(1)用事件格语法角色统一分流人/物/地/抽象四维度;
(2)在稳定事件 JSON 上做时间排序(单章成功,跨章机制可行但不稳)。**进行中。**

## 一、格角色四维度统一分流(物品/地点优化的全局解法)

### 问题
物品 pass 的错(ch44):医生(人)→物品、特别囚室(地点)→物品、长春刺杀案(抽象)→物品、
"藤原香子的榻榻米"带人名前缀。本质是"维度边界"问题——人/地/抽象混进物品。

### 解法:事件格语法角色 = 维度分类器(prompt: event_free_roles_v2)
不给候选,让事件抽取标清每个动作的格角色,**角色位置天然决定实体维度**:
- agent / patient(施事/受事的人) → **人物**
- instruments(被持握/使用/穿戴/摆弄的物理实体) → **物品**
- location(容纳动作发生于其中的空间) → **地点**
- 四格都不入(只能被谈论/思考,不可触摸/不能行动/不能容纳动作) → **抽象,排除**

### 验证(ch44/doc3/doc4/doc5 ×3)
- **维度过滤极可靠**:医生/特别囚室/长春刺杀案/夜莺(组织)/枫(代号)/克强 全部正确落选 instruments。
  长春刺杀案三章全部落选所有格角色(确认抽象)。地点(办公室/得胜楼/车内)稳定落 location 不混物品。
- **前缀脱落**:"藤原香子的榻榻米"→格角色只取核心物"榻榻米",无前缀。
- **特别囚室在 ch44 落选所有格角色是对的**:它只在台词里被提及,无动作发生于其中,本就不是本章实体。
- **局限:高精度低召回**(同 participants):instruments 只稳定抓"被显著操作"的核心物(烟/丝袜),
  背景物(办公桌/门/椅子)±1 摆动或漏。

### 统一架构(与人物线完全对称)
| | 高精度锚(事件格角色) | 高召回池(独立pass) |
|---|---|---|
| 人物 | participants | 人物pass |
| 物品 | instruments | 物品pass |
| 地点 | location | 地点pass |
格角色一个 pass = 人/物/地三维高精度锚 + 抽象过滤 + 维度纠错依据。
配独立 pass 补召回;独立pass抽的候选若在格角色里属别的维度或四格都不入→剔除/修正。

## 二、事件时间线

### 关键教训:别让模型吐全局 story_order
首次尝试让模型直接给每个事件 story_order(全局故事序号)——**失败**:模型小数乱凑(S9.5/S9.65)、
两跑逻辑完全不同、把当下对话误标闪回。全局时序排序是"全局推断"任务,模型做不稳。

### 成功方法:模型标局部信号 + 算法拼全局(prompt: event_free_roles_v4_sync)
模型只标【局部、有锚】的信号(它能做稳):
- is_flashback:该事件是否回忆/闪回内容。
- abs_time:绝对时间词,逐字摘原文(昨晚/一个多月前/今天中午),宁缺勿编。
- sync_with:跨场景同时标记("就在X的时候")——填别处同时情形的简述。**替代正则**(非正则、靠语义)。
算法做【确定性】:聚组(连续事件成组,flashback/gap 开新组,组内锁死)→ 选主线人物(participants最多)
→ 闪回组按时间锚归位(有"过去"锚才提前,无锚原位标记,不编造)。

### 单章验证(doc4):成功
doc4 场景2回忆段正确标 is_flashback + abs_time"一个多月前"→ 聚成闪回组 →
确定性重排把闪回组提前到当下组之前,**组内顺序保持不变**。闭环成功。

### 跨章拼接(doc3华剑雄线 + doc4地下党线):机制可行,端到端不稳
- **共享人物锚点:不可靠**。doc3∩doc4 人物=空集——平行线索人物常不重叠(谍战文各方各自行动),
  不能靠共享人物对齐。
- **sync_with 同时标记:最可靠的跨章锚**(作者显式声明两线同时)。一次验证中两跑都正确抓到
  doc4开头"就在华剑雄在得胜楼被丁墨村灌酒的时候"→ 钉到 doc3 得胜楼场景。
- **但 sync_with 召回不稳**:重抽 doc4 时该字段时有时无(全章可能仅开头一句,低频信号,模型易漏)。
  导致跨章端到端拼接时断时续。
- **abs_time 做粗对齐兜底**:昨晚/一个多月前/今天中午 可推大致先后。

### 跨章结论
跨章完整时间线**机制成立但当前不能可靠自动完成**,瓶颈=sync_with 召回不稳。
规律重现:局部稠密信号(格角色/is_flashback)模型标得稳;全局稀疏信号(sync_with,全章一两处)标得不稳。

### 下一步(未做)
- **sync_with 解耦**:像在场判定从事件抽取解耦那样,把"跨章/跨场景同时标记"做成专门的聚焦扫描pass
  (单独问"本章开头有无'与此同时别处…'交代句"),而非混在事件流顺带标(顺带易漏),提高召回稳定性。
- 更细的绝对时间解析(一个多月前 vs 三年前的先后、插入精确位置)。
- 多人时间线 + 共享事件锚点的完整拼接(待 sync 稳定后)。
