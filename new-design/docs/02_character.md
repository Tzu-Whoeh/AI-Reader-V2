# 人物分析(两-pass 架构)· 说明文档

中文叙事文本人物分析的完整方案,与物品分析同构:Pass1 识别人物 + 共指归并,Pass2 在固定人物清单上标注人物关系,合并管道复用物品那套(锚点校验 + id 引用校验)。

## 文件清单

| 文件 | 作用 |
|---|---|
| `人物分析_Pass1_人物识别.txt` | Pass1 提示词:识别人物 + 共指归并(本名/绰号/尊称/自称)+ 分配 id |
| `人物分析_Pass2_关系标注.txt` | Pass2 提示词:在 Pass1 固定清单上标四类人物关系,带方向和 confidence |
| `物品分析_合并管道.py` | 合并脚本(与物品分析共用,把 item 换成 character、part_of 换成 relations 即可) |

## 数据流

```
原文 ──Pass1──> {characters:[{id,name,aliases,role,confidence,...}]}
                      │
原文 + 人物清单 ──Pass2──> {relations:[{from_id,to_id,relation_type,label,confidence}]}
                      │
              merge + 校验(锚点 + id引用完整性)
```

## 七类人物关系(relation_type)

关系是**有向**的,from_id → to_id:

1. **social**(社会/隶属):上下级、同僚、同盟、敌对、合作。上下级方向为 上级→下级(如 老头子→华剑雄)。
2. **kin**(亲属/拟亲属):父子、义父子、兄弟、夫妻。方向 长辈/尊位→晚辈。
3. **affective**(情感/私人):双方**相互**构成的私人情感关系——情人、肉体关系、结义。注意只标双向关系,单向情绪归 attitude。
4. **event**(事件性,谁对谁做什么):一个角色对另一个实施的具体行为。方向 施事者→受事者(如 武田→那女人,刑讯)。
5. **allegiance**(立场伪装):A 对**清单中另一具体人物**表面如何、实则如何(如 华剑雄→武田,表面朋友实为监视)。**只标人对人**;"某人对某阵营的真实归属"(如华剑雄潜伏于76号)不用本类,记在 Pass1 的 role 里。
6. **awareness**(知情度):A 对 B 的真实身份/秘密的知情程度(如 颜雨→华剑雄,不知其秘密使命)。方向 认知主体→被认知对象。
7. **attitude**(态度/情绪指向):A **单方面**对 B 的主观情绪态度——紧张、敬畏、轻蔑、戒备等。方向 情绪持有者→情绪对象。与 affective 区别:attitude 单向(对方未必有对应情绪),affective 双向。同一人对不同人可有不同态度,各标一条。

同一对人之间允许多条不同类型关系(如华剑雄与武田:social 上下级 + allegiance 表面朋友 + attitude 觉得好笑)。

## id 引用机制(防脑补)

Pass2 只能引用 Pass1 清单里已有的 id,**机制上无法凭空创造不存在的人物**。合并时校验 from_id/to_id 必须指向有效 id,否则该关系作废。两次测试 id 引用完整性均 ✓。

## 调用配置

两 pass 同配置:`huihui_ai/Qwen3.6-abliterated:35b`,`temperature 0.12`,`format:"json"`,`num_ctx 8192`,`think:false`,`stream:false`。

## 验证结论

- **Pass1**:文本 A(10人)、文本 B(6人)均 id 连续唯一;核心共指正确(华剑雄←华处长/剑雄君/老子、老头子←周老板、武田←武田君);义父正确独立标 medium 不误并。
- **Pass2**:文本 A 标出 6 条关系、文本 B 标出 8 条,四类齐全;核心关系方向正确(老头子→华剑雄上下级、武田→无名女人刑讯、华剑雄→颜雨肉体关系);**id 引用完整性两次全 ✓**。
- 人物关系 Pass2 质量高于物品关系 Pass2 —— 因人物关系在原文多有明确动作/称谓依据(派遣、刑讯、肉体关系都明写),不像物品容器关系依赖模糊空间推断。

## 已知局限(不粉饰)

0. **新增三类(allegiance/awareness/attitude)方向可能反转**:这三类的方向约定较抽象(谁是情绪/认知/伪装主体),35b 语义判断准确但方向偶有标反(如"华剑雄对老头子紧张"被标成 老头子→华剑雄)。复核时重点核对这三类的方向。关系**是否存在**和**类型**通常是对的,只是箭头方向需校。

1. **social / event 边界模糊**:"雇佣司机"这类既可视为社会关系(主仆)又可视为事件(叫车),模型偶有归错类。不影响关系本身的识别,只是 type 标签。
2. **叙事对比误判为关系**:如原文"连丁主任进出都要检查"是用丁主任作对比,模型偶尔把它当成华剑雄与丁主任的真实关系(已标 medium)。
3. **同一对人关系略碎**:同一场互动(如一起去喝酒)可能被拆成多条 event。允许但偏冗。
4. **Pass1 残留**:跨段隐性等同(大使是否即裴仁基,原文无明说)模型会摇摆;偶发幻觉别名(如"华老板")——合并管道的锚点校验会兜底剔除。

**总原则**:关系方向与类型信 high、复核 medium;Pass1 的 name/alias 经锚点校验后可信;人工终裁存疑项。
