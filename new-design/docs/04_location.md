# 地点分析(两-pass 架构)· 说明文档

中文叙事文本地点分析,与人物/物品分析同构:Pass1 识别地点 + 共指 + 按尺度分类,Pass2 在固定地点清单上标四类空间关系。

## 文件清单

| 文件 | 作用 |
|---|---|
| `地点分析_Pass1_地点识别.txt` | 抽取地点 + 共指归并 + scale 分类(city/building/area/room)+ 分配 id |
| `地点分析_Pass2_关系标注.txt` | 在固定地点清单上标四类关系,有向,带 confidence |
| `物品分析_合并管道.py` | 合并脚本(共用;把 item 换成 location 即可) |

## scale 分类(空间尺度)

`city`(城市/地区,如长春/上海/虹口)、`building`(建筑或机构院落,如宪兵司令部/大使官邸)、`area`(建筑内区域,如花园/地下通道/走廊)、`room`(具体房间,如二楼办公室/七号刑讯室)。

## 四类地点关系(relation_type),均有向 from_id → to_id

1. **containment**(空间包含):A 在 B 内部。方向 被包含者→容纳者(如 二楼办公室→司令部)。**标直接上一层**,不越级(刑讯室→地下通道→司令部,不把刑讯室直连司令部)。
2. **adjacency**(方位/邻接):相邻或明确相对方位。label 写方位(如 草坪→刑讯室"在正上方")。
3. **movement**(移动路径):人物从 A 到 B。方向 起点→终点,按叙述顺序(办公室→刑讯室→地下通道→柳月仿)。
4. **remote**(远程叙事关系):非物理邻接、靠情节/回忆关联(如不同城市)。

## 关键设计:包含 ≠ 共指

Pass1 最易犯的错是把"上级场所内部的区域"当成上级的别名(如把"二楼办公室""花园"塞进"宪兵司令部"的 mentions)。本方案在 Pass1 明确:**上级场所与其内部区域是不同地点,各占 id**;它们之间的"谁在谁里面"由 Pass2 的 containment 标注。这样空间嵌套树是 Pass2 的输出,不污染 Pass1 的共指。

## id 引用机制(防脑补)

Pass2 只能引用 Pass1 已有 id,无法凭空创造地点。合并时校验 from_id/to_id 必须有效。测试中 id 完整性 ✓。

## 调用配置

两 pass 同:`huihui_ai/Qwen3.6-abliterated:35b`,`temperature 0.12`,`format:"json"`,`num_ctx 8192`,`think:false`,`stream:false`。

## 验证结论

- **Pass1**:文本B 10 项(修订后司令部不再吞下属区域,锚点全命中);文本A 城市/建筑识别准确,scale 分类对。
- **Pass2**:文本B 标出 11 条关系,containment 层级精准(直接上一层、不越级)、adjacency 抓住"刑讯室在草坪下方"的垂直关系、movement 还原移动动线、id 完整性 ✓。
- 地点关系是三套关系标注(人物/物品/地点)里**最准的**——空间关系在叙事文本中描写最具体明确,方向基本不反。

## 已知局限与后处理

1. **交通工具误收**:35b 会把"轿车/雪佛来"当成可进入的地点(因人物"钻进车里")。提示词已强调"交通工具是物品不是地点"仍偶发。
   → **强制后处理**:地点结果中 name 含 车/轿车/雪佛来 等交通工具词的,自动剔除(合并管道里加一条过滤即可):
   ```python
   VEHICLE_WORDS = ("车", "轿车", "雪佛来", "汽车")
   p1["locations"] = [l for l in p1["locations"]
                      if not any(w in l["name"] for w in VEHICLE_WORDS)]
   ```
2. **锚点**:全角数字(如"７６号")需逐字照抄,提示词已强调;仍用锚点校验兜底。
3. **remote 偶尔勉强**:机构关联(76号↔司令部)可能被标 remote,已用 medium 标出。

**总原则**:containment/movement 方向可信;name/mention 经锚点校验;交通工具用后处理硬剔除;medium 项人工复核。
