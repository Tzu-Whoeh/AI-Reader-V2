# 实体过度合并根治 · 封闭式清洗 + 模型复核归并

跨章实体归并里 **over-merge(过度合并)** 的成因、修复架构与方法论。配套模块:`entity_clean.py`、`entity_review.py`,集成在 `cross_chapter.run()`。本文是 `05_time_and_cross_chapter.md` 与 `06_entity_normalize.md` 的延续——前者讲跨章缝合骨架,后者讲"脏人名变体"归一,本文讲"不同人物被错误焊成一个"的反向问题。

## 一、问题形态

UI 上单个人物/物品节点的别名列表混入多个不同实体。典型现场:
- 人物"黎子午"别名含 刘大壮/吴四宝/黄克己 等多人,原文出处 1749 处(实为多人之和)。
- 物品"追魂章"别名含 骷髅头/两把交叉的利剑/正面/背面/厚厚的一本 等描述短语 + 处决令/照片/案卷/证章等不同物品,77 个 all_names 里 40 个是情节描述。

over-merge 比 under-merge(重名未合并)更刺眼,因为它把无关实体的引用全堆到一个节点,计数与关系全错。

## 二、三层根因

1. **归并算法过激(雪崩)**:`resolve_global_entities` 旧逻辑对"共享任一 merge_key"的节点对**无条件 `union`**,只在非同名时补记一条 ambiguity——但 union 已发生。叠加并查集传递性:A-B 并、B-C 并 ⇒ A-C 也连。一个泛称桥就能让整张图坍缩成一个超级簇。

2. **非人物维度零过滤 + mentions 误用**:物品/地点归并曾用 `mentions`(原文定位短语,**可带描述定语**)当归并键 *与* 展示别名;且非 characters 不过 `_is_merge_key`。于是"照片""文件""案卷"这种通用词字面相同就并,"那枚冷冰冰硬邦邦的徽章"这种描述短语进了 all_names。

3. **上游 pass1 偶发脏数据**:abliterated 模型抽取时偶发结构性错误——复合名("吴四宝和刘大壮"写成一条)、描述当名("吊着的女人""共党的区委书记")、别名串人(把别人的本名写进某人 aliases)。这些成为归并的错误桥。

> **核心判断**:单章产物大体是干净的(ch01 人物/物品 aliases 经核验无误),over-merge 主要诞生在**跨章归并阶段**;但上游 pass1 的偶发脏数据会喂给归并层当桥。所以修复要分"算法层"和"数据层"两路。

## 三、修复架构(四个动作)

### A. 归并算法收紧(确定性)

- **mentions 退出**:物品/地点不再用 mentions 当归并键或展示别名。每个章节局部节点新增 `aux` 集合容纳 mentions,**仅供原文定位**,既不进 merge_key 也不进 all_names。调用处 `alias_is_mentions=True` 标记。
- **全类型过滤**:所有实体类型的 merge_key 都过 `_is_merge_key`(剔除代词/泛称/绰号/单字/姓+通用职务),此前仅 characters 过滤。
- **exact-only 自动并(策略①)**:只有"双方本名逐字相同"才自动 `union`;所有别名/部分名称重叠一律**不并**,落 `ambiguities` 待复核。这是从"先并后记歧义"到"不确定就不并"的根本反转,符合 AGENT.md「uncertain→ambiguities,不硬猜」。
- **all_names 全类型清洗**:展示别名对所有类型做 `_display_names`(剔代词/贬称/纯泛称),此前仅 characters。

> 策略①会带来**欠合并**(同音异写"丁墨村=丁默村"、正式称谓"丁主任=丁墨村"被拆开)——这是刻意的保守,由下面的复核层消化。

### B. 模型复核归并(`entity_review.py`)

消化①产出的欠合并。对"本名不同但有重叠"的候选对,聚合双方身份画像(role/evidence),喂模型逐对判定是否同指,`same=True` 才合并。
- 判断交给模型(语义),执行用确定性并查集。
- 解决欠合并(丁组三写法合并)的同时**拒绝脏桥**(华剑雄是丁墨村下属→不并;周丽萍/周雪萍是姐妹→不并)。
- 按"本名对+画像指纹"缓存到 `.review_cache/review.json`,增量分析复用。

### C. 封闭式清洗(`entity_clean.py`)

治上游脏数据。对 **长度≥4** 的 name/alias 做**封闭二选一**判定(各 3 票多数):
- `multi?`——是否把多人并列写成一条(复合名)。
- `is_person?`——是否一个具体人物的称呼(否=情节/场景描述短语)。

name 命中 `multi=True` 或 `is_person=False` → 丢弃该条目;alias 命中 `is_person=False` → 从别名剔除。≤3 字免判直接留(复合名/描述短语几乎不可能这么短,省调用)。按词缓存到 `.review_cache/clean.json`。

### D. 集成与缓存(`cross_chapter.run`)

```
run(chapters, call_model=None, novel_root=None, use_llm=True):
    if llm: chapters = entity_clean.clean_chapters(...)   # 清洗在最前
    char_global = resolve_global_entities(...)            # ①exact-only
    if llm: char_global = entity_review.resolve_with_llm_review(...)  # 复核
    stitch_timelines(chapters, char_global, ...)          # 复核后再缝合
```

- **顺序铁律**:复核在 `stitch_timelines` **之前**完成,保证 timelines/relations 引用的 `global_id` 映射到复核合并后的最终节点(否则 relations 的 from/to id 错位)。端到端验证以 `relations id 越界==0` 为通过标志。
- **降级**:无 `call_model` 或 `use_llm=False` 时跳过清洗+复核,纯算法①运行,保证可离线。
- **缓存复用**:两层判定都按"词/对"缓存落盘,worker 增量聚合复用已判结果,仅新增词/对发起判定。

## 四、方法论:abliterated 模型怎么用才可靠

这是本次最重要的可复用结论。同一个 abliterated 模型,任务形式决定可靠性:

| 任务形式 | 例子 | 可靠性 |
|---|---|---|
| **封闭二选一** | "X 是否复合名?""A 与 B 是否同一人?" | **高**(抽样接近 100%) |
| **开放式挑错** | "这章人物清单有没有错?" | **低**(实测 76% 误报,意识流自我辩论) |
| **模糊归属** | "这个称呼合不合理地属于 A?" | **低**(放水严重,倾向判 True) |

实践要点:
- **逐项判,不要批量**:一次问多项时,项间互相干扰、自我辩论滚雪球。逐对/逐项判定稳定得多。
- **砍掉自由文本字段**:不给"reason"之类的写作空间,abliterated 会忍不住写思考链污染判断。只要 `{"same":true}` 这种最小结构。
- **多次投票抗抖动**:temp 即便压到 0.1,单次采样仍有尾部抖动;同一项判 3 次取多数(self-consistency),把偶发错误投票淘汰。
- **不确定就保守**:全部解析失败→默认不合并/不删除,绝不"猜一个"。

## 五、运维注意(踩过的坑)

- **worker 冷启动会拖死主分析**:把清洗+复核塞进 worker 的增量聚合后,首次缓存为空时 worker 反复触发全量 LLM 判定,并和主分析抢同一个 ollama,导致主分析卡死(done 计数不动、章目录不建)。**规避**:首次部署用离线脚本独立把 `.review_cache` 建满,再让生产走缓存(实测命中后整轮聚合 1 秒完成)。**待修**:worker 路径应在缓存未就绪时只跑算法、跳过 LLM,杜绝冷启动竞争。
- **请求级硬超时**:复核/清洗的 ollama 调用必须设 `socket.setdefaulttimeout` + urlopen timeout(实测某些词会触发模型长响应/疑似 hang,无超时会卡住整个后台任务)。
- **增量落盘**:判定缓存边判边写,卡死重启可续跑,避免重判。

## 六、已知局限 / 待办

- **≤3 字脏别名串人未根治**:清洗只判 ≥4 字,故"剑雄"(华剑雄简称被错挂到丁主任)、"黎座"等短串名仍残留。它们**不影响合并正确性**(复核不会因此错并),仅污染展示。尝试过"确定性筛嫌疑(别名==他人唯一专名)+ AI 判归属",但 AI 的"归属是否合理"判定放水严重(把明显该剔的"勾明在黎子午""周雪萍在周丽萍"判成保留),**不够可靠,未采用**。根治需更强的带上下文判定或更严的提示词设计。
- **泛称多节点(欠合并的另一面)**:"老头子"被抽成 8 个独立节点、"周老板"22 个——泛称被反复抽成独立人物,既是欠合并也污染别名。是否合并需区分"同一称谓指多个不同人"与"同一人被重复抽",属独立课题。
- **清洗的取舍**:描述当名被丢弃后,原文里**无本名、仅以描述指代的角色**(如"无名女人""受刑女人")会从人物图消失。这是"宁缺勿滥"的合理取舍,但需知晓。
- **`event_pipeline.py` 写死 `11434`**:与 `app.py` 的 `OLLAMA_URL`(默认隧道 18434)规范不一致,应改为读环境变量。
