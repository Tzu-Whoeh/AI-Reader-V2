# 中文叙事文本 · 场景拆分提示词 V4(定稿)

把一段中文叙事文本按"叙事单元"切分成若干场景,输出结构化 JSON。经 4 轮迭代 + 换文本泛化验证收敛。

---

## 一、调用配置

| 项 | 值 | 说明 |
|---|---|---|
| 模型 | `huihui_ai/Qwen3.6-abliterated:35b` | 36B MoE / Q4_K_M。明显优于同系列 27b(27b 倾向粗粒度归并、漏切) |
| 端点 | `POST http://127.0.0.1:11434/api/generate` | Ollama |
| temperature | `0.15` | 压制切点随机漂移的关键 |
| format | `"json"` | **必须**。从解码端强制 JSON,根除中文全角冒号导致的解析失败 |
| num_ctx | `8192` | ~1900 字文本足够;更长文本需上调 |
| stream | `false` | |
| think | `false` | 关闭思考链,省 token、加速 |

请求体示例:

```json
{
  "model": "huihui_ai/Qwen3.6-abliterated:35b",
  "prompt": "<下方提示词，{TEXT} 处替换为待拆分文本>",
  "stream": false,
  "think": false,
  "format": "json",
  "options": { "temperature": 0.15, "num_ctx": 8192 }
}
```

性能:单次推理约 5-9 秒(模型已驻显存)。

## 二、提示词正文(将 `{TEXT}` 替换为待拆分文本)

```
你是中文叙事文本的场景拆分助手。把【待拆分文本】按"叙事单元(scene/beat)"切分。

【粒度基准:中粒度】
目标是把一段文本切成"叙事功能不同的几大块"，通常 4-6 个。不要切到句子级。

一个场景在"叙事功能发生转折"处开始。四种功能:
- 现实叙述:当下正在发生的客观情节。
- 回忆:对过去事件的客观回放(有明确的过去时空)。
- 内心独白:人物当下的思考、盘算、情绪。
- 动作:人物当下一连串身体动作(可含多个连续动词)。

【该切】功能从一种变为另一种时切。例如:现实→回忆、回忆→内心独白、内心独白→动作。

【不该切 — 重要】
- 同一功能内部不要再切。一连串连贯动作("坐下→写字→拿糖→起身→检查枪→出门")属于同一个"动作"场景，整体算一段，绝不逐动作拆开。
- 同一段回忆里的多个细节(任务背景、人物介绍、事件经过)若时空连续，归为一个"回忆"场景，不要按话题拆碎。
- 判断依据是"叙事功能"，不是句子数或动词数。

【location】客观回放往事填往事地点(如"长春")；当下内心活动填人物当下所在地(如"办公室")；过渡或不确定填"未明"。不加括号。

【type】四选一原样输出，单值:现实叙述 / 回忆 / 内心独白 / 动作

【参考示范】(另一段文字的正确拆分，仅示范粒度与字段，不要套用其内容)
{"scene_count":3,"scenes":[
{"index":1,"title":"深夜归家","type":"现实叙述","location":"家门口","summary":"主角深夜回到家，发现门虚掩着。","start_text":"他摸黑爬上楼梯","end_text":"门竟然是开的"},
{"index":2,"title":"回想白天的争执","type":"回忆","location":"办公室","summary":"主角想起白天和上司激烈争吵的经过。","start_text":"白天那场争吵","end_text":"摔门而去"},
{"index":3,"title":"推门查看","type":"动作","location":"家中","summary":"主角掏出钥匙、推门、开灯、环视屋内。","start_text":"他握紧钥匙","end_text":"屋里空无一人"}]}

【输出格式 — 极重要】
- 只输出一个 JSON 对象，第一个字符是 { ，最后一个字符是 } 。
- 禁止 markdown 标记、禁止任何前后缀文字。
- 所有 JSON 结构符号(引号、冒号、逗号、花括号)用 ASCII 半角。中文全角标点只能出现在字段文字内容里，绝不用作 JSON 分隔符。
- start_text 取场景开头原文前 10-15 字，end_text 取结尾原文后 10-15 字，能唯一定位边界即可。

结构同示范。

【约束】按原文顺序切，不重排、不遗漏、不重叠；start_text/end_text 逐字摘自原文；不臆造。

【待拆分文本】
{TEXT}
```

## 三、输出结构

```json
{
  "scene_count": 5,
  "scenes": [
    {
      "index": 1,
      "title": "≤15字标题",
      "type": "现实叙述 | 回忆 | 内心独白 | 动作",
      "location": "叙事时空地点，或 未明",
      "summary": "1-2句概括",
      "start_text": "开头原文片段(用于在原文中定位边界)",
      "end_text": "结尾原文片段"
    }
  ]
}
```

`start_text` / `end_text` 是逐字原文锚点,下游可用 `str.find()` 在原文里精确定位每个场景的起止偏移,从而真正切开原文。这是防模型幻觉的关键设计。

---

## 四、验收标准

1. **合法 JSON**:能被 `json.loads` 一次解析(`format:json` 已基本保证)。
2. **中粒度**:场景数落在 4-6,不出现句子级碎切。
3. **type 单值**:四类之一,无复合标签(如"内心独白/动作")。
4. **不漏硬换场点**:钟声、"蓦然想起"、突然声响等明确的回忆/思考→动作切换点必须切开。
5. **不过切连贯段**:连续动作流、时空连续的回忆不被拆碎。
6. **锚点忠实**:`start_text`/`end_text` 逐字出自原文,可定位。

## 五、迭代历程(供调参参考)

| 版本 | 改动 | 结果 |
|---|---|---|
| V1 | 基础提示词 | 6 段,JSON 合法但 location 套错、粒度偏碎 |
| V2 | 区分 location 叙事时空、type 强制单值 | location 改善,但**中文全角冒号导致 JSON 解析失败** |
| V3 | 加 `format:json` + ASCII 标点硬约束 | JSON 稳了、提速到 5s,但**多次运行切点漂移大、连贯动作被过切** |
| **V4** | **加 few-shot 示范 + 反向"不该切"约束 + 锁定中粒度** | **稳定 5 段,3 次运行 type 序列一致、切点基本一致** |

**最有效的两招**:few-shot 示范(让模型看到正确粒度长什么样)、反向"不该切"约束(明确连贯动作/连续回忆不拆)。

---

## 六、测试结论

在两种结构迥异的文本上均通过:
- **回忆插叙型**(办公室回想往事):3 次运行均 5 段,type 序列完全一致,切点基本逐字相同。
- **单线推进型**(到达→下行→对话→离开):提示词原样不改,3 次均 5 段,前 3 个功能转折切点逐字一致;location 自动填出具体地点。

---

## 七、已知局限

- **无硬换场点的连续段**(如"进门→目睹→对话→离开"这类纯现实流)切点会在合理范围内小幅漂移,且 type 可能在"动作/现实叙述"间摆动。这是文本本身边界模糊所致,非提示词缺陷,人工扫一眼即可定夺。
- **长文本**:测试文本约 1300-1900 字。更长篇需上调 `num_ctx`,并建议分章送入,避免上下文过长导致后段切分质量下降。
- **批量使用**:建议保留 `start_text` 锚点做程序化原文切割后,对 `location` 字段和无硬换场点处的切点做轻量人工抽检。
