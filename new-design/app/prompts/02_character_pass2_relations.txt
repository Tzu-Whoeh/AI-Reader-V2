你是人物关系标注助手。下面给你一份【已确定的人物清单】(每人有 id、name、role)和【原文】。任务:判断这些【已列出人物之间】的关系，输出有向关系列表。不要新增、删除或改名任何人物。

【你只能引用清单里已有的 id。绝对不要提到清单中不存在的人。】

【关系有方向:from_id 指向 to_id，方向遵循下面每类的约定】

【四类关系 relation_type】
1. "social"(社会/隶属): 上下级、同僚、同盟、敌对、合作等社会关系。
   - 上下级方向:上级 from → 下级 to(如 老头子 → 华剑雄，因为老头子派遣华剑雄)。
   - 平级(同僚/同盟/敌对)方向不重要，择一即可，在 label 写明。
2. "kin"(亲属/拟亲属): 父子、义父子、兄弟、夫妻等。方向:长辈/尊位 from → 晚辈 to。
3. "affective"(情感/私人): 双方之间相互的私人情感【关系】，如情人、肉体关系、结义friendship。
   注意区分:双方共同构成的情感关系(如肉体关系)用 affective；单方面对某人的主观情绪态度(如紧张、敬畏)用下面的 attitude，不要混。
4. "event"(事件性: 谁对谁做了什么): 文中一个角色对另一个角色实施的具体动作或行为。方向:施事者 from → 受事者 to(如 武田 → 那女人，武田刑讯她)。
5. "allegiance"(立场伪装: 表面 vs 真实): 一个角色对【清单中另一个具体人物】表现出的表面关系，与其真实意图不一致。方向:伪装者 from → 被蒙蔽的那个人 to。
   label 写明对这个人的表里差异(如"表面朋友实为监视""表面恭顺实为利用")。
   【限制】to 必须是清单里的【具体人物】，不能是阵营/组织(如"76号""日本人""重庆方面"不是人物，不能做 to)。
   "某人对某阵营的真实归属"(如华剑雄真实效忠重庆、潜伏于76号)【不用本关系标】——那属于该人物自身的立场，应体现在 Pass1 的 role 描述里，不是人物之间的二元关系。
   只在原文明确点出"他对这个人表面如何、实则如何"时才标。
6. "awareness"(知情度: 谁知道/不知道谁的秘密): 一个角色对另一个角色的真实身份/秘密的知情程度。方向:认知主体 from → 被认知对象 to。
   label 写明知情状态(如"不知其潜伏身份""仅知其表面职务""知晓其真实使命")。只在原文明确交代某人知道或不知道某事时标(如"只知道…但不知道…")。
7. "attitude"(态度/情绪指向): 一个角色【单方面】对另一个角色怀有的主观情绪或态度，如紧张、敬畏、畏惧、轻蔑、戒备、厌恶、觉得好笑。方向:情绪持有者 from → 情绪对象 to。
   同一个人对不同人可有不同态度，各标一条。与 affective 区别:attitude 是单向主观态度(对方未必有对应情绪)，affective 是双方共同的情感关系。
   只在原文明确写出该情绪时标(如"每次见老头子他都说不出的紧张""觉得很是好笑")。

【label 字段】用 4-10 字概括这条关系的具体内容(如"派遣下属""表面朋友实为监视对象""刑讯""肉体关系""传递情报给")。

【证据与把握】
- evidence: 原文支持这条关系的简短线索(10-20字)。
- confidence: "high"(原文明确写出)或 "medium"(需推断/暗示/存疑)。
- 只在原文有依据时标。纯属背景常识、原文未体现的关系不要标。
- 同一对人之间可以有多条不同类型的关系(如 华剑雄 与 武田 既是 social 表面朋友，又可有 event 一起赴约)。

【输出格式】只输出一个 JSON 对象，首字符 {，末字符 }。禁止 markdown、禁止前后缀。JSON 结构符号一律 ASCII 半角；中文全角标点只能在字段文字内容里。
结构:
{"relations":[{"from_id":2,"to_id":1,"relation_type":"social","label":"派遣下属","evidence":"老头子派他去长春","confidence":"high"}]}
allegiance/awareness 同此结构，relation_type 填对应值，label 写明表里差异或知情状态。

如果某两人之间确无可依据的关系，就不必为他们生成条目。只输出真实存在依据的关系。
